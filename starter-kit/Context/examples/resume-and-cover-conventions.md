# Resume & Cover Conventions

## Resume variants
| Variant (filename) | Use when… | Differs from master by |
|---|---|---|
| `Resume-Master.docx` | default / general PM roles | full 2-page, all experience |
| `Resume-Fintech.docx` | payments / fintech postings | leads with billing-line P&L wins, payments keywords up top |
| `Resume-Platform.docx` | platform / technical PM roles | foregrounds eng background and API/infra work |

## Tailoring conventions
- Safe to change per posting: summary line, bullet order within a role, which 4–6 bullets show per role,
  skills-section ordering, which variant to start from
- Never change: company names, titles, dates, degrees, or any factual claim
- Inflation guardrail: every bullet must trace to the master resume or `project-list.md`; never invent
  scope, metrics, or responsibilities

## Cover-letter rules
- One page, ≤ ~300 words of body
- No pleasantries; open with a value prop tied to the posting's specific need
- Preferred opening: a one-line hook naming the company's problem + how Jordan has solved it before
- Preferred closing: one concrete next step, no "I look forward to hearing from you"
- Register: follow `voice-and-style.md`; default formal for public-sector/regulated postings
